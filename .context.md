## Conventions & Patterns

### Naming Conventions
- **Classes**: PascalCase (e.g., `OrganizationService`)
- **Methods**: PascalCase (e.g., `GetByIdAsync`)
- **Properties**: PascalCase
- **Private fields**: `_camelCase` (e.g., `_repository`)
- **Database tables**: PluralPascalCase (e.g., `Organizations`)

### Async/Await Pattern
- All I/O operations must be `async`/`await`
- Methods return `Task<T>` or `Task`
- Suffix with `Async` (e.g., `GetAsync`)

### Entity Framework Core
- Use **Repository Pattern** to abstract DbContext
- Implement **Unit of Work** for transactions
- Use **DbContext Scoping** (scoped lifetime in DI)
- Enable **Change Tracking** for auditing
- Use **Soft Deletes** (IsDeleted flag) not hard deletes

### API Responses
- Return `Result<T>` wrapper with:
  - `Data: T` (success payload)
  - `Error: string?` (error message)
  - `IsSuccess: bool`
- Use **Problem Details** (RFC 7807) for errors
- HTTP status codes: 200 (OK), 201 (Created), 204 (No Content), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found), 500 (Internal Error)

### Logging
- Use `ILogger<T>` injected via DI
- Log levels: Debug, Information, Warning, Error, Critical
- Include context (user ID, organization ID, request ID)
- Structured logging (key-value pairs)

## Important Rules

1. **Never expose DbContext outside Data layer**
2. **All DTOs** (Data Transfer Objects) separate from Entities
3. **All operations** must be idempotent where possible
4. **Soft deletes** by default; hard deletes require special justification
5. **No business logic in controllers** - delegate to services
6. **Circular dependencies** are forbidden - use interfaces
7. **Configuration** via `appsettings.json` and environment variables
8. **No raw SQL** unless performance-critical (use EF Core LINQ)
9. **Transactions** only where data consistency requires it
10. **API versions** in URL if breaking changes occur: `/api/v1/`
